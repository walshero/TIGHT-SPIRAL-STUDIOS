# ZIP AUDIT — "backups or originals?"
*2026-07-11 · answered by opening them, not guessing*

## THE ANSWER: **BOTH — and that's the dangerous kind.**

They are **snapshot exports** — a whole project folder zipped at a moment in time. Most
of what's inside is duplicated elsewhere. **But not all of it.** A handful of files exist
**only** inside these archives, and some of them matter a great deal.

**A zip that is 95% redundant and 5% unique is not a backup. It is a hiding place.**

---

## NEW CAPABILITY BUILT THIS SESSION

`list_zip_contents` — a Zapier code action that reads a zip's **central directory** via
an HTTP Range request (last 64KB) and returns the file table **without downloading the
archive.** Works on files up to 300MB.

This closes a real blind spot: 40MB of archives were unopened because base64-decoding
them would blow the tool-result ceiling — the *same* wall that froze the Confluence
trunk, pointed the other direction. **Now I can see inside a 16MB zip in one call.**

**Known limit:** it fails on **zip64** archives (`Matt's Writing…zip`, 11.8MB, errored).
That format needs a different parser. Named, not hidden.

## THE REAL LESSON

The zips are **the archaeology of the drift.** Every one is a moment where the studio
tried to save itself by copying everything, sideways, into a new container.

**Ten `tight-spiral-*.zip` snapshots inside one zip** is not a backup strategy — it is
the same panic, ten times. And it is precisely what a **pointer** architecture exists to
end: *canon is an address, not a copy.* You cannot zip your way out of ambiguity. Each
snapshot **adds** a candidate for "which one is real."
