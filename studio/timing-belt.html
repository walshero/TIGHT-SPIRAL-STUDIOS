The Timing Belt

The part that keeps memory, the shelf, and Drive turning in phase.

A sync copies state between two places. A timing belt keeps every rotating part turning in phase so the engine doesn't tear itself apart. When it slips, parts collide. That is the studio's failure mode: memory says one thing, the shelf another, Drive a third — out of phase, so work collides and evaporates.

The honest limit. A single offline file cannot reach out and write to Drive, the Project shelf, and memory on its own. There is no button that makes a change propagate across all three. The Belt is a protocol Claude runs against a ledger — it finds every phase-slip and names the correcting move. It does not auto-wire the surfaces. Promising the wire is the exact confident-but-wrong failure the Belt exists to prevent.

The three surfaces

Everything the studio knows lives on one of three surfaces. Phase-slip is when they disagree.

Memory what Claude's memory claims exists — loads every chat, can go stale.
Shelf the Project files (durable, phone-saved — the working truth).
Drive Claude_files — the dated archive, reachable via the Drive connector.

Running the Belt — the protocol

When you say run the belt, Claude walks these five turns. Each has a fixed job so it can't be done freehand.

  1. Inventory each surface List what each surface holds right now — memory's claims, the shelf's files, Drive's files. Read, don't assume.
  2. Byte-check the shelf Every file named "done" gets a real size on disk (the File-Save Verify Floor). Zero-byte = a ghost, not a file.
  3. Find the phase-slips Cross the three lists. Flag: files in one surface missing from another; conflicting versions; stale memory claims; ghosts.
  4. Name the correcting move For each slip, the single move that re-phases it — and who makes it (Claude reads/builds; Matt phone-saves; the Drive push).
  5. Emit the ledger + Carry-Out One reconciled ledger + the paste-into-next-session line, so the fix lands instead of evaporating.
