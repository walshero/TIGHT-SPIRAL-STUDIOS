Tight Spiral Studios · Production System

The Runbook

From a concept in your head to a thing you can play — run by you, one step at a time. Every step tells you what to hand Claude, what Claude does, and how you know it's done.

How this works

Three workshops. A concept goes through them in order. Each one has a gate — you don't move on until the gate is passed. Open a step, copy its prompt, paste it to Claude, get the output, check the gate, move on.

  1. Workshop 1 — Concept Validationdoes the idea hold up before anyone builds?
  2. Workshop 2 — Spec Buildturn the concept into a buildable blueprint
  3. Workshop 3 — Prototype Buildget something playable, checked, on your phone
  4. The Agentswho each voice is and what they're guarding
  5. Decision Logso nothing gets re-litigated
Workshop 1

Concept Validation

You have an idea. Before a single line of spec gets written, the idea gets pressure-tested against the Four Freedoms, the learning principles, and the trauma-informed / accessibility floors. Gate: a locked one-page Concept Brief.

1 Pitch the concept ▸

You do

Say the idea out loud, however rough. The prompt below frames it so Claude knows to play the Concept Architect — clarify, don't build yet.

Copy → paste, then add your pitch at the bottom 
Act as the CONCEPT ARCHITECT for Tight Spiral Studios.

Your job right now is NOT to build anything. It is to take my raw pitch and turn it into a clear, testable concept. Do this:

1. Restate the concept in two sentences: what the player does, and what they learn by doing it.
2. Name the learning objective(s) in plain language.
3. Name the core loop: learn a move -> hold it -> use it -> get better at it.
4. Flag anything ambiguous you need from me before this can be validated. Ask me at most 3 questions.

Do not write a spec. Do not design mechanics in detail. Keep it to one page.

MY PITCH:
[ write your idea here ]

Done when

Claude hands back a two-sentence concept, named objectives, the loop, and at most 3 questions. You answer the questions in the same chat.

Next: Step 2 — run it past the principle guards.
2 Principle & safety pass ▸

You do

In the same chat, run the concept past the three guards that can HALT it: Four Freedoms, trauma-informed, accessibility-as-novelty.

Copy → paste (same chat as Step 1) 
Now convene three guards on the concept above. Each gives a verdict: PASS, FIX (with the specific fix), or HALT (with why). Be concrete, not generic.

1. FOUR FREEDOMS GUARD (Osterweil). Does the concept actually deliver freedom to experiment, freedom to fail safely, freedom to try on identities, freedom of effort? Name where each one lives in the concept, or where it's missing.

2. TRAUMA-INFORMED GUARD. Scan for predator/prey framing, violence or weapon similes (e.g. "sniper," "cracks like a gunshot"), "sense weakness" language, anything that could land badly. Flag every instance, even in metaphor.

3. ACCESSIBILITY-AS-NOVELTY GUARD. Does the novelty survive for a player who switches modes (screen reader, reduced input, etc.)? Where would the playful surprise DIE for them, and what's the equivalent novelty in another affordance (sound, haptics, timing)?

End with a single line: CONCEPT IS CLEAR TO PROCEED / CONCEPT NEEDS REWORK.

Done when

All three return PASS or actionable FIX. Any HALT means you revise the concept and re-run this step. Don't carry a HALT forward.

Next: Step 3 — lock the brief.
3 Lock the Concept Brief ▸

Claude does

Produces the one-page brief as a clean file you keep. This is the only thing that moves to Workshop 2.

Copy → paste (same chat) 
Write the locked CONCEPT BRIEF as a single-file document I can save. One page. Sections:

- Title + one-line hook
- What the player does / what they learn (2 sentences)
- Learning objective(s)
- Core loop
- Motivation archetypes this should support (collector / competitor / socializer / achiever / meter-watcher) and one note each on how the loop reskins for them
- Four Freedoms: one line each on where it lives
- Guard sign-offs (carry forward the verdicts from the last step)
- OPEN QUESTIONS that the spec stage must resolve

Then present it as a file.

Done when

You have a saved Concept Brief file with guard sign-offs. Gate passed. This file is the input to Workshop 2.

Workshop 2

Spec Build

The brief becomes a blueprint a builder could follow without guessing: mechanics, mastery scaling, content list, the authoring fields, and the accessibility rendering. Gate: a spec that passes the efficacy and buildability check.

4 Draft the spec ▸

You do

Start a fresh chat. Paste your Concept Brief file, then the prompt below.

Copy → paste (new chat, with the Brief attached) 
Act as the GAME DESIGNER + INTERACTION DESIGNER for Tight Spiral Studios. The attached Concept Brief is locked — do not reopen its decisions.

Write a SPEC a builder could follow without guessing. Sections:

1. MASTERY LADDER. For each move/competency the player learns, define 3 levels: weak (just met it), working (used it a few times), mastered (full power). Define what unlocks each level and what each level lets the player DO. The card-too-heavy-to-swing principle: a half-learned tool is visibly limited.

2. CORE LOOP, STEP BY STEP. Every screen the player sees in one run, in order. One decision per screen.

3. CONTENT LIST. Everything that needs to be authored (clips/prompts/moves), as a table the author fills in. Keep it lightweight — buttons and fields, no code.

4. AUTHORING FIELDS. The actual form a teacher fills to make one instance of this game. List every field and its type.

5. MOTIVATION RESKIN. How the same loop renders for each archetype from the brief.

6. ACCESSIBILITY RENDERING. For every novel/playful element, the equivalent in another affordance. This is a floor, not a toggle.

Keep it buildable. Flag anything that would force custom code per-instance as a RISK.

Done when

You have a spec with all six sections and any code-per-instance risks flagged.

Next: Step 5 — the efficacy & buildability gate.
5 Efficacy & buildability gate ▸

You do

Same chat. This is the gate that saves your iteration time later — it catches "this won't actually teach" and "this will break on a phone" before anything is built.

Copy → paste (same chat) 
Convene four guards on the spec. Verdict each: PASS / FIX / HALT.

1. LEARNING SCIENTIST. Does the mastery ladder actually teach the objective, or just gate content? Is there real freedom of effort — can a player invest at their own level? Where's the assessment hidden inside play (not a quiz bolted on)?

2. PROJECT MANAGER (scope guard). Does anything here require custom code per instance? Is the authoring layer truly lightweight? Name the shippable MVP — the smallest version that proves the concept. Cut everything else to a "later" list.

3. PLATFORM & AFFORDANCE TECHNOLOGIST. Will this run as a single offline HTML file on phone width? Flag anything that needs a server, a build step, or heavy assets.

4. INCLUSIVE IDENTITY LEAD. Any assumed gender, family shape, body, or skin coding anywhere in the content or examples? Human figures stay neutral kraft tone.

End with: SPEC READY TO BUILD MVP / SPEC NEEDS REWORK. If ready, restate the MVP in 3 bullets.

Done when

All four PASS, and you have a 3-bullet MVP definition. HALT → revise spec, re-run. Gate passed when you can name the MVP.

6 Lock the spec ▸

Claude does

Folds the gate fixes in and produces the final spec file with the MVP called out at the top.

Copy → paste (same chat) 
Produce the LOCKED SPEC as a single file. Put the 3-bullet MVP at the very top, boxed. Fold in every FIX from the gate. Append the guard sign-offs. Append a "LATER (not in MVP)" list so scope creep has a home. Present it as a file.

Done when

Saved spec file with MVP boxed at top. This is the only input to Workshop 3.

Workshop 3

Prototype Build

The MVP gets built as a single offline HTML file, checked for bugs and device-breakage, then handed to you to actually play. Gate: it runs on your phone, and you've played it — not read about it.

7 Build the MVP ▸

You do

Fresh chat. Paste the locked Spec file, then the build prompt.

Copy → paste (new chat, Spec attached) 
Build the MVP from the attached spec — the 3 boxed bullets only, nothing from the LATER list. BUILD IT mode: don't re-ask, don't reopen locked decisions, don't optimize before it ships.

Hard requirements (Tight Spiral house floor):
- Single file, fully offline HTML. No external calls.
- Accessibility floor: large type, one decision per screen, big tappable buttons, full keyboard nav, visible focus rings, reduced-motion respected, scroll resets to top on every screen change, nothing hidden mid-screen.
- Cut-paper craft look where there's art: flat layered shapes, edge-shadow underneath, matte grain, real value range, no brown-on-brown mud. Suggest, don't simulate.
- No emoji. No violence/weapon imagery even in similes.
- Works at phone width.

Then present it as a file and tell me, in 3 lines, what to try first when I open it.

Done when

You have an HTML file and a 3-line "try this first."

Next: Step 8 — the bug & device check, before you waste a play session on a broken build.
8 Bug & device check ▸

You do

Same chat. This catches the "is it even on the page / does it break the device" layer you named, so you only spend real play-time on builds that run.

Copy → paste (same chat) 
Before I play, run the build check. For each item: PASS / FAIL (with the exact fix). Then apply all fixes and re-present the file.

1. Does it render — is every screen actually on the page, nothing blank or off-canvas?
2. Keyboard: can I reach and trigger every control with Tab + Enter/Space? Visible focus everywhere?
3. Does any state dead-end — a screen with no way forward or back?
4. Phone width: does anything overflow, overlap, or get cut off?
5. Reduced motion: does it still work with animation off?
6. Does the core loop complete start to finish without a console error?

Note: you are the build's eyes for code. For how it LOOKS, I'm the eyes — I'll tell you after I play.

Done when

All six PASS and you have a re-presented file. Now — and only now — you play it.

Next: Step 9 — you play, then the visual/feel verdict.
9 Play it, then the feel verdict ▸

You do

Open it on your phone. Play the whole loop. Then report back — you're the eyes for how it looks and the gut for how it feels. The prompt below turns your reactions into fixes via the right guards.

Copy → paste (same chat), filling in your reactions 
I played it. Here's what happened: [ what worked, what felt off, what didn't read, what to change ]

Run the Tight Spiral loop on what I reported: diagnose the root cause, fix it, AND fix the sibling issues it implies — don't just patch the one thing I named. Route my visual notes through the Visual Critic, my "what is that" notes through the Reads-As Legibility Critic, and my "this felt flat / promised motion that didn't happen" notes through the Motion & Interaction Animator.

Re-present the file. Tell me what changed and what you fixed that I didn't mention.

Done when

The build does what your gut wanted. Loop Steps 8–9 as many times as it takes — but each loop must ship a new file, never just talk about one (the scope guard enforces this).

When it's good enough to show someone else, it's your MVP. Log the decision (below), then either start Workshop 1 on the next concept, or add a "LATER" item back in for a v2.
Reference

The Agents

Each "agent" is one voice with one job and the authority to stop the line. You're playing them through Claude until you automate it. The ones that can HALT are the ones worth never skipping.

+ Who's who ▸

Workshop 1 — Concept

  • Concept Architect — turns a pitch into a testable concept; does not build.
  • Four Freedoms Guard (HALT) — experiment / fail / identity / effort must all be present.
  • Trauma-Informed Guard (HALT) — no predator/prey, no weapon similes, nothing that lands badly.
  • Accessibility-as-Novelty Guard (HALT) — the playfulness must survive a mode switch, in some affordance.

Workshop 2 — Spec

  • Game Designer + Interaction Designer — mechanics, mastery ladder, lightweight authoring fields.
  • Learning Scientist (HALT) — does it teach, or just gate content?
  • Project Manager (SCOPE HALT) — names the MVP, kills code-per-instance, sends extras to LATER.
  • Platform & Affordance Technologist — single offline file, phone width, no server.
  • Inclusive Identity Lead (HALT) — no assumed gender/family/body; neutral kraft figures.

Workshop 3 — Build

  • Builder — single offline HTML, house floor, BUILD IT mode.
  • Build Checker — renders? keyboard? dead-ends? overflow? reduced motion? loop completes?
  • Visual Critic (HALT) — judges the render; you are the eyes, your "doesn't pass" is the HALT.
  • Reads-As Legibility Critic (HALT) — can a stranger name it at a glance on a phone.
  • Motion & Interaction Animator — catches promised-but-static motion.
Keep the panel as small as the job allows. A concept needs the Workshop 1 four. A finished build doesn't need the Concept Architect again. Convene the guards the work actually touches.
Reference

Decision Log

The thing that keeps you from re-litigating settled calls. Every time a decision locks, write one line here. When an agent tries to reopen it, you point at the line.

+ How to keep it ▸

One line per locked decision

Format: date — what was decided — why — do not reopen.

  • Keep it in the Concept Brief and Spec files themselves, at the bottom.
  • When you start a new chat, the locked files carry the log with them, so Claude sees what's settled.
  • If an agent reopens a logged decision, that's a scope-guard failure — paste the line and move on.
Copy → paste any time a decision locks 
Add this to the decision log at the bottom of the current file, then re-present the file:

[ date ] — [ what we decided ] — [ why ] — DO NOT REOPEN.
